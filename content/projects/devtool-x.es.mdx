## El problema

En equipos distribuidos, el clásico "en mi máquina funciona" tiene una causa concreta: las configuraciones de entorno divergen sin que nadie lo note. Variables de entorno, versiones de herramientas, flags de feature — cada desarrollador tiene su propio estado y no existe una forma simple de comparar, detectar diferencias o reproducir el entorno de un colega.

Los proyectos grandes acumulan `.env.example` desactualizados, wikis con instrucciones viejas y onboardings que llevan días. El problema no es la intención — es la falta de herramienta.

## La solución

DevTool X expone un CLI (`dtx`) que captura el estado del entorno local — variables, versiones de runtimes, configuraciones de herramientas — y lo sincroniza con un servidor central. Desde la UI web podés ver en tiempo real qué tiene cada miembro del equipo, comparar dos entornos lado a lado y detectar exactamente qué difiere.

El onboarding de un proyecto nuevo pasa de "seguí este doc de 30 pasos" a `dtx sync` y una pantalla que te muestra qué te falta configurar.

La arquitectura es deliberadamente simple: Bun como runtime para el CLI y el servidor (velocidad de arranque y compatibilidad nativa con TypeScript sin build step), tRPC para la comunicación tipada entre CLI y API, y React para la UI.

## Stack y decisiones

**Bun** fue la apuesta central del proyecto. El CLI necesitaba arrancar en menos de 100 ms — Node.js con ts-node no lo lograba sin compilación previa. Bun ejecuta TypeScript nativo y el tiempo de arranque es imperceptible.

**tRPC** eliminó la necesidad de mantener un contrato de API separado. El CLI y el servidor comparten los tipos directamente — si el servidor cambia un endpoint, el CLI falla en compilación, no en runtime.

**React** para la UI sin framework adicional. La UI es una SPA simple servida por el mismo proceso de Bun — sin Next.js, sin overhead. La complejidad no lo justificaba.

No hay base de datos persistente por default: el servidor guarda el estado en memoria y puede configurarse con un backend de persistencia (SQLite, PostgreSQL) si el equipo lo necesita.
