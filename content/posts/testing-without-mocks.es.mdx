Hay una conversación que se repite en casi todos los equipos con los que trabajo: "tenemos 90% de cobertura de tests y aun así todo se rompe en producción".

La cobertura no miente. Pero tampoco dice la verdad completa.

## El problema con los mocks

Los mocks son útiles. No voy a decir que hay que eliminarlos. Pero cuando se convierten en la estrategia por defecto para aislar dependencias, empezamos a testear una versión imaginaria del sistema en lugar del sistema real.

Considerá este escenario clásico: tenés un servicio que busca usuarios en una base de datos. Para testearlo en aislamiento, mockeas el repositorio. El test pasa. Todo bien. Después en producción, el query tiene un problema con la paginación que el mock nunca habría detectado porque el mock hacía exactamente lo que vos le programaste que hiciera.

**Cuando mockeas una dependencia, estás testeando que tu código funciona correctamente con una dependencia que se comporta perfectamente según tus suposiciones. Eso no es un test, es una tautología.**

## Tests de integración: la alternativa pragmática

Un test de integración, en mi definición de trabajo, es cualquier test que ejercita más de una unidad real del sistema. No necesita ser complejo. No necesita un ambiente de staging completo. Necesita ser honesto.

Para trabajar con TypeScript en modo integración sin mocks pesados, suelo apoyarme en tres estrategias:

- **Bases de datos reales en memoria** (SQLite para tests que normalmente usan Postgres, por ejemplo con `better-sqlite3` o directamente con `pg` apuntando a un contenedor efímero en CI).
- **Implementaciones alternativas del puerto**, no mocks sino stubs reales que implementan la interfaz pero se comportan de forma predecible.
- **Test containers** cuando la infraestructura es parte del contrato.

El punto es que el código que se ejecuta en el test se parece mucho más al código que se ejecuta en producción.

## Un ejemplo concreto en TypeScript

Supongamos que tenemos un caso de uso que registra un nuevo usuario y envía un email de bienvenida. La versión con mocks pesados es la más común, pero veamos la alternativa:

```typescript
// Puerto (la interfaz del dominio)
interface UserRepository {
  save(user: User): Promise<void>;
  findByEmail(email: string): Promise<User | null>;
}

interface EmailService {
  sendWelcome(to: string, name: string): Promise<void>;
}

// Caso de uso
class RegisterUser {
  constructor(
    private readonly users: UserRepository,
    private readonly email: EmailService,
  ) {}

  async execute(input: { name: string; email: string }): Promise<User> {
    const existing = await this.users.findByEmail(input.email);
    if (existing) {
      throw new Error(`Email ${input.email} is already registered`);
    }

    const user = User.create({ name: input.name, email: input.email });
    await this.users.save(user);
    await this.email.sendWelcome(user.email, user.name);
    return user;
  }
}

// Stub real para tests — no es un mock, es una implementación alternativa
class InMemoryUserRepository implements UserRepository {
  private store = new Map<string, User>();

  async save(user: User): Promise<void> {
    this.store.set(user.id, user);
  }

  async findByEmail(email: string): Promise<User | null> {
    for (const user of this.store.values()) {
      if (user.email === email) return user;
    }
    return null;
  }
}

class CapturingEmailService implements EmailService {
  readonly sent: Array<{ to: string; name: string }> = [];

  async sendWelcome(to: string, name: string): Promise<void> {
    this.sent.push({ to, name });
  }
}

// Test de integración
describe('RegisterUser', () => {
  it('saves the user and sends a welcome email', async () => {
    const users = new InMemoryUserRepository();
    const emailService = new CapturingEmailService();
    const registerUser = new RegisterUser(users, emailService);

    const result = await registerUser.execute({
      name: 'Juan Silva',
      email: 'juan@example.com',
    });

    // Verificamos el estado real del repositorio
    const saved = await users.findByEmail('juan@example.com');
    expect(saved).not.toBeNull();
    expect(saved!.id).toBe(result.id);

    // Verificamos que el email se "envió" con los datos correctos
    expect(emailService.sent).toHaveLength(1);
    expect(emailService.sent[0]).toEqual({
      to: 'juan@example.com',
      name: 'Juan Silva',
    });
  });

  it('throws when the email is already taken', async () => {
    const users = new InMemoryUserRepository();
    const emailService = new CapturingEmailService();
    const registerUser = new RegisterUser(users, emailService);

    await registerUser.execute({ name: 'Primero', email: 'dup@example.com' });

    await expect(
      registerUser.execute({ name: 'Segundo', email: 'dup@example.com' }),
    ).rejects.toThrow('already registered');
  });
});
```

La diferencia clave está en `InMemoryUserRepository` y `CapturingEmailService`. No son mocks creados por un framework que interceptan llamadas. Son implementaciones reales del puerto, escritas para el contexto de test, que se comportan de forma determinista sin depender de magia de reflection o de la firma exacta de los métodos para interceptarlos.

## Por qué esto importa en términos de arquitectura

Cuando usás mocks al estilo `jest.fn()` o `vi.fn()` para reemplazar métodos en lugar de interfaces, el test está acoplado a los detalles de implementación. Si cambias el nombre interno de un método, el test rompe aunque el comportamiento externo sea idéntico. Eso es ruido, no señal.

Las implementaciones alternativas, en cambio, están acopladas a la interfaz del puerto, que es exactamente el contrato que debería importarnos. Si la interfaz cambia, los tests tienen que adaptarse —y eso es correcto, porque cambió el contrato.

Además, tener `InMemoryUserRepository` como clase real en el proyecto tiene un beneficio secundario: podés usarla en el desarrollo local para correr la aplicación sin infraestructura. Un adaptador de desarrollo que es también el adaptador de test. Dos pájaros de un tiro.

## Cuándo sí uso mocks

No soy dogmático. Uso mocks cuando:

- La dependencia externa es difícil o costosa de controlar en tests (un proveedor de pagos externo, un servicio de SMS).
- Quiero verificar que una función fue llamada un número específico de veces con parámetros específicos y eso es parte del comportamiento observable del sistema.
- Estoy testeando una unidad verdaderamente aislada y el comportamiento de la integración ya está cubierto en otro nivel.

La clave es no usar mocks como primera respuesta al problema de "esto tiene dependencias". La primera respuesta debería ser: **¿puedo hacer una implementación alternativa honesta de esta dependencia?**

## Cerrando

Los tests deberían darte confianza de que el sistema funciona. Eso implica que tienen que ejercitar el sistema de una manera que se parezca a cómo el sistema va a funcionar en producción.

Los mocks heavy son una salida rápida que compra velocidad a corto plazo y vende confianza a largo plazo. Las implementaciones alternativas son más trabajo inicial, pero construyen una suite de tests que te dice algo real sobre el software.

Y al final, eso es lo que queremos: tests que, cuando pasan, signifiquen algo.
