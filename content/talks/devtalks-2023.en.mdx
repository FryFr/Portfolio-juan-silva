## Abstract

There is a quiet trap in software teams: well-built systems that nobody can explain. The code works, the tests pass, but the knowledge lives inside one person's head. Architecture, when misunderstood, becomes a hoarded secret instead of a shared language.

This talk proposes thinking about architectural decisions as pedagogical acts. Every layer, every abstraction, every naming convention is an opportunity to communicate intent. A well-designed system should teach the next developer why it was built that way — without needing a 45-minute meeting.

## Core Ideas

- **Code structure is documentation.** If you have to explain why something lives where it lives, the structure is lying.
- Premature abstractions aren't just a performance problem — they're a comprehension problem. When you hide complexity before you understand it, you multiply it.
- Screaming Architecture isn't a trend — it's a bet on domain readability over framework readability.
- Onboarding someone new is the real stress test for your architecture. If it takes them weeks to understand the system, something in the design is failing.

## Who This Talk Is For

For tech leads and senior developers who already know how to build systems that work, but want to build systems that others can understand, maintain, and improve with confidence.
